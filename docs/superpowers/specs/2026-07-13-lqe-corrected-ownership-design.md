# LQE `corrected` 写入权修复设计

## 目标

彻底消除未经人工确认的新译名进入 `corrected` 的可能，同时保留标准 `_corrected.xlsx` 输出。

本次修改不改变问题判定、问题类型、严重程度、分组方式、扣分公式或通过阈值；不重新处理既有任务，也不改写既有输出文件。

## 根因

旧流程同时要求：

1. 子任务模型发现问题后必须填写完整 `corrected`。
2. 子任务模型遇到术语表缺词、错误、超长或多种译法时不得自行决定最终译法。
3. 合并脚本直接采用子任务模型提供的 `corrected`。

这使“发现问题”和“可以安全修改”无法分开。子任务模型即使没有可靠译法，也必须填写 `corrected`；合并脚本随后把这段内容当成可执行修改。

## 核心规则

### `corrected` 只有一个写入者

`corrected` 只由 Python 合并程序生成。主 agent、子任务模型和各检查模块都不得输出或改写该字段。

检查模块只输出：

- 问题说明；
- 问题类型和严重程度；
- 可选的局部修改；
- 是否需要人工确认。

若检查模块结果出现顶层 `corrected`，结构检查立即失败，后续计分、报告和导出全部停止。

### `corrected` 的唯一含义

`corrected` 表示：已经通过程序检查、可以写入 `_corrected.xlsx` 的完整建议译文。

- 没有可执行修改：`corrected = null`。
- 存在一个或多个可执行修改：程序以原译为基础应用修改，生成完整字符串。
- 只有需要人工确认的问题：保留原译，`corrected = null`。
- 同一行同时存在可执行修改和需要人工确认的问题：只应用可执行部分，需要人工确认的文字保持原样。

## 检查模块输出

每条结果使用以下结构：

```json
{
  "id": 127,
  "issues": [
    {
      "category": "Terminology",
      "severity": "Major",
      "comment": "术语表内容与项目要求不一致，需要人工确认。",
      "needs_confirmation": true,
      "edit": null
    },
    {
      "category": "Punctuation",
      "severity": "Minor",
      "comment": "标点格式不符合项目要求。",
      "needs_confirmation": false,
      "edit": {
        "from": "错误片段",
        "to": "正确片段",
        "evidence": null
      }
    }
  ]
}
```

整句重写同样作为一条修改提交：`from` 为完整原译，`to` 为完整建议译文。检查模块仍不填写 `corrected`。

### 前后示例

普通标点问题中，检查模块不再提交完整译文，只提交 `“แบลนดาโร” → "แบลนดาโร"`。Python 检查后生成完整建议译文 `การ์ดหายาก "แบลนดาโร"`，结果与原流程一致。

`深蓝鲸` 的原译是 `ธาลูน`，术语表译法超过字符限制。检查模块只记录问题并设置 `edit: null`；Python 生成 `corrected: null`，不得产生新的专名译法。

`奇丽花印象-鞋子` 的原译是 `สไตล์ฟลอโรรา- รองเท้า`。检查模块把专名问题记录为需要人工确认、不给修改；另把缺少空格记录为 `ฟลอโรรา- รองเท้า → ฟลอโรรา - รองเท้า`。Python 最终生成 `สไตล์ฟลอโรรา - รองเท้า`：安全修改生效，专名保持原样。

## 修改检查

Python 程序在生成 `corrected` 前逐条检查修改：

1. `from` 必须能在原译中唯一定位，或明确给出出现位置。
2. 修改不得破坏变量、标签、换行标记和受保护文本。
3. 需要人工确认的问题不得携带修改。
4. 专名修改必须与一条已确认的术语表译法完全一致，并记录对应术语。
5. 两条修改覆盖范围重叠且结果不同，不应用冲突部分；该部分改为需要人工确认。
6. 不重叠的安全修改继续应用，不受同一行其他问题影响。

任一检查失败时，不静默采用模型结果。结构错误使当前检查文件失败；内容冲突只阻止冲突部分，其他安全修改继续处理。

`needs_confirmation` 只是检查模块提供的信息，不是最终决定。即使检查模块填写 `false`，Python 程序仍会按上述规则重新检查；专名修改缺少已确认术语依据时一律不执行。

## 术语处理

术语表项目分为“已确认”和“参考”两类。没有明确标记的专名按“参考”处理。

只有“已确认”术语可以驱动自动修改。以下情况只写入 LQE 报告，不生成新译名：

- 术语表缺词；
- 术语表值为空、为占位内容或明显错误；
- 术语表译法违反字符限制；
- 项目规则要求更新旧译，但没有提供已确认的新译法；
- 原译与参考术语不同；
- 不同检查模块给出不同专名译法。

标准 LQE 流程不再承担新专名命名。若用户需要新译名，必须作为独立任务处理，确认后写回术语表，再进入下一次 LQE。

## 计分

计分逻辑保持原样：

- 沿用现有问题类型、严重程度和扣分公式；
- 沿用现有重复段分组和计数方式；
- 沿用现有通过阈值；
- `needs_confirmation` 和是否存在修改均不改变分数；
- 本次修改只决定建议能否进入 `corrected`。

## 合并流程

1. 各检查模块输出问题和局部修改。
2. 结构检查拒绝任何由模型填写的顶层 `corrected`。
3. 合并程序按段汇总问题，保留现有问题归属和计分输入。
4. 修改检查逐条判断哪些修改可以执行。
5. 程序把可执行修改应用到原译，生成唯一的 `corrected`。
6. 计分程序按原有规则计算。
7. 报告程序写出 LQE 报告。
8. 导出程序只把程序生成的 `corrected` 写入 `_corrected.xlsx`。

## 输出

标准任务只生成：

- `*_lqe.xlsx`
- `*_corrected.xlsx`

不再生成额外确认表。需要人工确认的内容放在 LQE 报告中。

LQE 报告使用普通用词：

- 原译
- 建议译文
- 处理方式
- 建议修改
- 需要人工确认
- 仅提醒
- 无需修改
- 已保护，不修改

`_corrected.xlsx` 保持原文件结构，不新增处理状态列。

终端输出只显示：

- 建议修改数量
- 需要人工确认数量
- 保持原译数量
- 已保护数量

## 命名清理

删除旧的人类可见流程术语，统一使用：

- 检查模块
- 术语表
- 已保护
- 建议修改
- 需要人工确认
- 项目负责人
- 确认规则

项目确认记录文件改名为 `confirmed_rules.md`；检查模块目录和命令同步改为普通命名。旧模块名和旧命令不保留。

`corrected` 作为标准机器字段和 `_corrected.xlsx` 文件名保留，但不再作为人类可编辑的状态或建议栏。

## 既有任务

不读取旧检查模块输出，不迁移旧状态，不重新处理原始 Excel，也不改写已生成文件。本设计只对修改后的新任务生效。

## 验收用例

1. `深蓝鲸`：原译保持不变；术语表超长问题进入报告；不得生成新的专名译法。
2. `伊里斯`：原译保持不变；术语表错误进入报告；不得生成另一种拼写。
3. `星光狮`：原译保持不变；报告说明缺少已确认的新译法；不得自行创造译名。
4. `魔草巫灵`、`奇丽花`：只有术语表译法已确认时才能自动替换。
5. `花衣蝶`：不同检查结果不得产生两套可执行译名。
6. 混合问题行：标点或语法修改正常进入 `corrected`，未确认专名保持原样。
7. 检查模块输出顶层 `corrected`：结构检查失败。
8. 未确认专名修改：修改检查失败，不进入 `corrected`。
9. 修改冲突：冲突部分不应用，非冲突部分正常应用。
10. 计分结果：同一组 `errors` 在修改前后得到完全相同的分数。
11. `_corrected.xlsx`：只写入程序生成的安全修改，结构与输入文件一致。
12. 报告、终端、表头和文件名不出现旧的人类可见流程术语。
