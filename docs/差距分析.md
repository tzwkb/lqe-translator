# 本地化质量评估 (LQE) Skill 差距分析

> 基线：ISO 5060:2024 / MQM-Core / SAE J2450 / LISA / TAUS DQF + 游戏本地化特有维度
> 对象：现有 ZH→EN 游戏 LQE Skill（燕云十六声/WWM）
> 格式：每项 现状 → 行业标准要求 → 差距与风险 → 优先级

---

## GAP-01｜类型学顶层维度与 MQM/ISO 5060 不对齐

**现状**：16 类归入 6 父类（`CATEGORY_PARENT`）：Accuracy / Fluency / Style / Terminology / Design / Other。`Locale convention`/`Length`/`Markup` 归入 Design；`Culture specific reference` 归入 Accuracy。

**行业标准要求**：MQM-Core 与 ISO 5060:2024 共享同一套 **7 个一级维度**：Terminology、Accuracy、Linguistic Conventions、Style、Locale Conventions、**Audience Appropriateness**、Design and Markup。每错误经 Decision Tree 归入唯一维度。

**差距与风险**：
- 缺失独立的 **Audience Appropriateness（受众适配）**顶层维度——游戏合规/文化敏感/分级相关的误译无归属处，被迫塞进 Accuracy 或 Other，弱化游戏最关键的合规信号。
- `Culture specific reference` 归到 Accuracy 与 MQM 不符：MQM 中文化/世界观矛盾属 **Verity/Audience Appropriateness** 而非 Accuracy（Accuracy 限源↔译语义保真）。
- `Linguistic Conventions`（含 Punctuation/Spelling/Grammar）被本 skill 命名为 Fluency——可接受（DQF 调和版用 Fluency），但与 ISO 5060 命名不一致，跨标准报告时映射困难。

**优先级**：**P1**（维度缺失影响合规信号，但可通过新增父类增量修复）

## GAP-02｜Verity 父类死代码 / 映射不一致

**现状**：`_PARENT_REMAP` 引用 `"Verity"` 父类，但 `CATEGORY_PARENT` 中无任何类别映射到 Verity——死代码/不一致。同时 `Culture specific reference` 实际被映射到 Accuracy。

**行业标准要求**：DQF-MQM 调和版将 **Verity**（culture-specific reference、与作品世界观矛盾文本）列为独立顶层组；映射表须自洽——每个被引用的父类必须有实际子类别归属，每个子类别必须落入已定义父类。

**差距与风险**：
- 死代码暴露设计意图与实现脱节：原设计应有 `Culture specific reference → Verity`，但落地时被错接到 Accuracy，留下悬空的 `_PARENT_REMAP["Verity"]`。
- 风险：后续若有人按 `_PARENT_REMAP` 添加 Verity 子类，权重/归一化静默错误；当前 `Culture specific reference` 错误地继承 Accuracy 的 1.5 权重而非 Verity 应有的处理。
- 维护陷阱：两张映射表（`CATEGORY_PARENT` / `_PARENT_REMAP`）真相不唯一。

**修复方向**：建立 `Culture specific reference → Verity`（或并入新增的 Audience Appropriateness），删除或激活死分支，二选一使映射自洽。

**优先级**：**P0**（已知 bug，映射不自洽，静默错误风险，修复成本低）

## GAP-03｜严重度乘数与权重设计偏离指数间距

**现状**：严重度 `Neutral=0 / Minor=1 / Major=5 / Critical=10`；权重二值制：多数"重要"类=1.5，其余=1.0；4 类（Terminology/Untranslated/Markup/Length）强制 Major。

**行业标准要求**：
- MQM 默认严重度乘数 **0/1/5/25**（指数间距），Critical=25 而非 10；指数间距使 Critical 在评分中占据主导，匹配其严重性。
- MQM Error Type Weight 默认=1，可针对内容上调（如法律内容上调 Accuracy）；权重应反映维度对该内容类型的关键度，而非二值近似。
- J2450 采用按类别差异化权重（serious 上限 5、SP/PE/ME 更低），证明扁平权重并非最佳。

**差距与风险**：
- **Critical=10 vs MQM=25**：本 skill 中 Critical 仅为 Major（5）的 2 倍，远低于 MQM 的 5 倍（25/5）。Critical 信号被压缩——损坏占位符/攻击性误译的惩罚不足以拉开与普通 Major 的差距。注：本 skill 另设 Critical 自动 Fail 门可部分补偿，但分数本身失真。
- **二值权重 1.5/1.0 过粗**：无法表达"游戏中 Terminology 应上调 ETW"的细粒度（证据明示游戏术语应上调 ETW）；marketing 桶的 Style 与 UI 桶的 Locale 用同一套权重。
- **强制 Major 合理但僵硬**：Length 强制 Major 在 narrative 桶（允许 transcreation 长度偏离）可能误伤——见 GAP-05 内容画像。

**优先级**：**P1**（影响分数区分度与排序，但有自动 Fail 门兜底；调参即可改善）

## GAP-04｜评分归一化缺失 per-1000-词，阈值单一硬编码 98

**现状**：`score = max((1 − ΣL / wordcount) × 100, 0)`，`wordcount` 首轮锁定；阈值**单一硬编码 98**，全内容类型通用。

**行业标准要求**：
- 全框架共享"加权惩罚和 ÷ 词数"内核，且**统一归一化至每 1000 词**（MQM RWC=1000 / LISA 50/1000 / DQF per-1000），使 5 词 UI 串与 200 词任务日志可比。
- **小样本低置信**：MQM Multi-Range——单 UI 串（<~250 词）线性分数应视为低置信，转而依赖规则/风险检查，不直接打分。
- **内容画像分级阈值**：DQF 明示 pass/fail 阈值随内容类型/受众/时效变化，可按目标语言覆写。

**差距与风险**：
- **未归一化到 per-1000**：`ΣL / wordcount` 是 per-word 密度，方向对但量纲非行业标准 per-1000；更关键是**短串方差爆炸**——5 词串里 1 个 Major（5×1.5=7.5 惩罚）→ `1 − 7.5/5 = −0.5` → clamp 到 0 分，单错即归零，过度惩罚短 UI 串。
- **无小样本低置信机制**：违反 MQM Multi-Range，短 UI 串本应转规则判定却仍线性打分，分数不可信。
- **单一阈值 98 跨内容类型一刀切**：与 DQF content profiling 直接冲突。98（≈ per-word 0.02 惩罚预算）对 narrative/transcreation 桶过严（该桶允许授权偏离、意义等价优先于字面），对 UI 功能串可能过松（功能错误应近零容忍 + 自动 Fail）。无按桶差异化 metric/权重/通过线。
- **wordcount 首轮锁定**：apply-fixes 迭代中若文本长度变化，分母固定可能使后续轮次分数轻微失真（次要）。

**优先级**：**P0**（短串归零是评分核心缺陷，直接导致大量 UI 串误判 FAIL；阈值单一与内容画像根本冲突）

## GAP-05｜缺失内容画像分桶（content profiling）

**现状**：单一评分模型 + 单一阈值 98，不区分内容类型；内置风格指南全局统一。

**行业标准要求**：DQF content profiling 是落地核心——按内容桶配置不同 metric、严重度权重、通过线：
- **UI/utility**（菜单/系统/错误消息）：Accuracy + Locale + Design(Length/Truncation) 严格。
- **Item/skill tooltip + 战斗数值**：Terminology + 数字一致性严格。
- **Brand/marketing**：Fluency + Style + Verity，语气近零容忍。
- **Narrative/dialogue**：Fluency + Style + Register + Verity，允许 transcreation 授权偏离，意义等价优先于字面 adequacy。

**差距与风险**：
- 游戏文本天然跨四桶（菜单/技能 tooltip/剧情对话/商城文案），单一模型必然在某些桶失准：narrative 桶被 98 阈值和字面 Accuracy 误伤；UI 桶的功能错误未获足够严格对待。
- 无桶感知 → 无法实现"marketing 语气近零容忍""narrative 意义等价优先"等差异化标准。
- 与 GAP-04 阈值问题同源：content profiling 是阈值差异化的上位机制。

**优先级**：**P1**（结构性增强，依赖输入侧能标注/推断内容桶；P0 的阈值问题可先用粗粒度桶缓解）

## GAP-06｜缺失游戏特有确定性检查（pre-check 覆盖不全）

**现状**：pre-check 覆盖：含中文→Untranslated、破折号、颜色标签 `#G/#C/#E` 数量、变量 `{}`/`%s` 数量、`\n` 数量、非CJK长度 1.5×、千位分隔符、术语表命中缺译。

**行业标准要求**（§2.2 确定性维度 + §4 自动化检查清单）：游戏 LQE 应覆盖以下纯文本可判定项。逐项对照现有 pre-check：

| 检查项 | 行业要求 | 现状 | 差距 |
|---|---|---|---|
| 占位符**顺序** | 源↔译占位符顺序一致（不仅数量） | 仅查数量/缺失多余 | **缺顺序校验**——重排序占位符（多语序常见）数量相同但语义错位，漏检 |
| 文本截断/长度 per UI | 译文对比**字段宽度上限**（固定宽 UI） | 仅查 1.5× 源文比例 | **缺 per-field 宽度上限**——无 UI 字段约束，固定宽控件溢出漏检 |
| 复数/性别一致 | 复数形式、性别一致（语法一致性） | 无 | **完全缺失**——EN 复数/代词一致未检（映射 J2450 WS） |
| 文件内术语一致性 | 相同源不同译 / 相同译不同源（intra-file） | 仅查术语表命中 | **缺一致性交叉检查**——同一文件内同源异译漏检（DQF Inconsistent use） |
| 双空格/首尾空格 | 双空格、首尾空格、悬空标点 | 无 | **完全缺失**——whitespace 类确定性错误未检 |
| 语气/语域 (register) | 敬语/正式度一致（RPG/MMO 对话关键） | 无（且属判断类，AI 侧可补） | 确定性侧难，但 register 一致性可部分规则化（如 you/thou 混用） |
| 数字一致性 | 源↔译数值/范围/单位一致（战斗数值关键） | 仅查千位分隔符格式 | **缺值一致性**——技能数值译错（如 100→1000）漏检，游戏 Critical |
| 未译/漏译扩展 | 目标=源、空译、TM 不当精确匹配 | 仅查 target 含中文 | 含中文外的未译（如英文源原样复制）漏检；空译未检 |

**差距与风险**：占位符顺序、数字值一致性、复数/性别、文件内一致性、whitespace 五项为**纯文本可确定判定**且多为游戏 Critical/Major 级，当前完全或部分缺失，导致本应在 pre-check 拦截的高危错误下放到 AI 判断（非确定、有方差）或漏检。数字值错误（技能伤害 100→1000）是游戏功能性 Critical，漏检风险最高。

**优先级**：**P0**（数字值一致性 + 占位符顺序属游戏 Critical 漏检；复数/性别/whitespace/文件内一致性为 P1 增量）

## GAP-07｜重复错误折扣未实现（死列）

**现状**：xlsx 报告含 `repeated error` 列但**硬编码为 0**，未实现重复错误折扣逻辑。

**行业标准要求**：
- 证据核实——具体折扣**数值**业界无公开标准（不应臆造数字）。
- 可迁移机制为 **J2450 两条元规则**：① 跨类别归属取标准列表中**最早**类别（WT→SE→OM…）；② 严重度存疑取**重**（默认 SERIOUS）——配合 MQM Decision Tree 单维度归属，从源头消除重复计数与跨轮方差，使 LLM 评分**确定化**。

**差距与风险**：
- 死列误导：报告呈现"已考虑重复错误"的假象，实际恒为 0。
- 真正价值不在"折扣数值"而在**去重机制**：同一术语在文件内重复误译 N 次，当前按 N 次全额计惩罚，可能使长文件分数被单一系统性错误反复拉低；而行业做法是单维度归属 + 早类别胜出消除重复计数。
- LLM 评分确定性缺口：无 Decision Tree 式单维度强制归属，同一错误跨轮可能归入不同类别，分数有方差。

**修复方向**：要么实现"同源同译重复错误按首次全额、后续折扣/单次计"的去重规则（需与用户确认折扣策略，不臆造数值），要么移除死列。引入早类别胜出 + 严重度存疑取重的确定化元规则。

**优先级**：**P1**（死列误导 + 系统性错误重复计惩罚；但无业界标准数值，需先与用户确认策略）

## GAP-08｜范围边界声明清晰但需显式契约化

**现状**：范围局限已声明——"纯文本 LQE：不涉及功能性 bug、视觉/截断渲染、配音/对口型、年龄分级合规等"。

**行业标准要求**：游戏 QA 四分类——Linguistic / Cosmetic / Functional / Compliance。LQE（文本语言层）≈ Linguistic，而：
- **Cosmetic**（截断/溢出/换行渲染）需**运行时/视觉**验证——文本侧仅能"预测"风险（长度比/字段上限），不能确诊。
- **Functional**（占位符损坏、热键冲突、技能数值）——**部分可文本侧确定**（占位符/数值/未译），部分需运行时（热键冲突、变量运行时解析）。
- **Compliance**（文化敏感/分级）——**部分可文本侧判断**（攻击性文本、世界观矛盾），部分需法务/分级机构。

**差距与风险**：
- 声明正确但**边界粒度过粗**：把"文本可预测的 Cosmetic/Functional/Compliance"与"必须运行时的部分"一刀切排除，可能放弃文本侧本可拦截的高价值检查。
  - 例：截断**风险**（长度超字段上限）文本侧可预测 → 应纳入（GAP-06）；截断**渲染确诊**才需排除。
  - 例：技能数值错误（Functional）文本侧可确定 → 应纳入（GAP-06）；攻击性误译（Compliance）文本侧可判断 → 应纳入（GAP-01 Audience Appropriateness）。
- 风险：过宽的排除声明成为"不实现游戏特有检查"的挡箭牌，与 GAP-06 的可确定项产生张力。

**修复方向**：将范围边界从"按四分类整类排除"细化为"**按可文本确定性**划界"——文本侧可确定/可预测的子项纳入 LQE 并标注置信度（确诊 vs 风险预测），仅运行时/视觉/法务依赖项显式移交 LQA/功能测试/合规审查。明确 LQE↔LQA 交接契约。

**优先级**：**P2**（声明已存在且方向正确，属边界精细化；实质漏检由 GAP-06 承载，此项为契约化与职责划清）

## 优先级汇总

| 优先级 | Gap | 一句话 |
|---|---|---|
| **P0** | GAP-02 | Verity 死代码/映射不自洽，静默错误风险，低成本修复 |
| **P0** | GAP-04 | 短串归零 + 无 per-1000 归一化 + 单一阈值 98 与内容画像冲突 |
| **P0** | GAP-06 | 数字值一致性 + 占位符顺序等游戏 Critical 确定性检查缺失 |
| **P1** | GAP-01 | 缺 Audience Appropriateness 顶层维度，文化/合规信号无归属 |
| **P1** | GAP-03 | Critical=10≪MQM 25，二值权重过粗，区分度不足 |
| **P1** | GAP-05 | 无内容画像分桶，narrative/UI 桶用同一标准必失准 |
| **P1** | GAP-07 | 重复错误折扣死列 + 系统性错误重复计惩罚 + 无去重元规则 |
| **P2** | GAP-08 | 范围边界应从"整类排除"细化为"按文本确定性划界" |

**P0 修复序**：GAP-02（孤立 bug，最低成本）→ GAP-06（补游戏 Critical 确定性检查）→ GAP-04（重构归一化与阈值，工作量最大，需配合 GAP-05 内容画像）。
