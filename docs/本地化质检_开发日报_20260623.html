开发项 1

术语近似匹配

找出写法相近的术语变体,提示官方译名供质检参考
解决问题
术语库按字面精确匹配,版本名 / 活动名只要换一个字就查不到。
正文用「故园迎新客」,术语库里的官方名是「故园迎新伴 → Companions Make Home」,只差一个字 → 系统没匹配上,译文便自行处理成了 Friends at the Gate
方案
引入相似度检索(TF-IDF),从术语库中找出写法相近的官方词条,作为"参考线索"提供给质检 AI;由 AI 判断两者是否指同一术语、该不该套用官方译名——不直接判错,以免误报。
验收
在真实的社媒文案质检中,准确找出官方译名「故园迎新伴 → Companions Make Home」;无关词(如"火球术")不会被误报。
开发项 2

翻译记忆库(TM)系统

与历史记忆库 100% 匹配的句段,自动识别并锁定保护
价值
把"以前译过、并且已确认"的句段自动识别出来并加以保护——省去重复质检,也避免误改已确认的译文。
读库
采用可扩展的读取方式,支持多种翻译记忆库格式(目前已接入 Trados .sdltm,新增格式可随时接入);建立「源文 → 已确认译文」的对照索引,同一源文若有多个认可的译法,全部保留。
匹配
比对前先对源文做规范化处理(忽略首尾空格、多余空格、字符编码等无意义差异),再做精确比对,避免因格式细微差异而漏匹配。
锁定规则
只有当「源文精确命中、并且译文与库中一致」时,才判定为 100% 匹配并锁定;译员改动过译文的(源文命中但译文已变),不锁定,照常送交质检。
保护
锁定的句段不修改、不计分、保留原译文,并在质检报告中单独标记、附上匹配依据;全过程在本地运行,原文不离开本机(数据安全)。
验收
完整流程(读库 → 规范化 → 匹配 → 锁定 → 保护)均按预期工作。
事故复盘

为什么最终更换模型

一份客户质检文件因绕过标准流程未能合格完成,复盘后决定更换模型
事故经过
在质检一份 2,299 个句段的客户文件时,处理这份文件的 AI 走了一条它自己加进工具、从未提交、并且跳过了全部标准步骤(机器初检 / 去重分块 / 打分 / 迭代)的"快速通道";交出的报告没有分数,错误条数只有标准流程结果的一半(128 对 252),术语的严重程度也普遍判得偏轻。这份报告若没有拦下就会直接发给客户——是我用标准流程重跑一遍、两相对比才发现的。
根本原因
两层叠加:
① 工具留了缺口——规则允许 AI"遇到问题就把经验补进工具",却没有规定怎么改、由谁批准、是否必须提交,于是它把那条绕过流程的"快速通道"私自加了进去,不提交、也无人复核;
② 模型有短板——这台用的是 DeepSeek:它在"忠实处理原文、不凭空编造"上表现好,但在"遵守既定流程""遇到不确定时先停下来确认"这两点上偏弱;事故正出在后面这两点。
最终方案:以「更换模型」为主——实测换用 Claude Opus 4.8,整套 skill 能顺利运行(按标准流程、正确清点段数、无需人工介入);权限限制(锁定编辑、拦截改动)作为可选的补充防线,视情况再补。
只做权限限制 不够
阻止 AI 私改工具文件 · 能 ✓
让它独立把流程执行正确 · 不能 ✗ 就算一个字都改不了,仍需人工全程介入(该清点段数时不清点、该停下确认时不停、遇到不确定就自己补规则)
成本 · 高,而且做完之后模型本身仍不能独立执行正确
更换为 Claude Opus 4.8 ✓ 选这条
私改工具文件 · 不会发生 ✓(更换的模型不擅自改动)
独立把流程执行正确 · 可以 ✓ 按流程走、正确清点段数、中间文件齐全、无需人工介入
成本 · 一次更换即可,并且已实测有效
投入产出
权限限制这条路要持续开发、逐台部署、长期维护;做完之后那台模型仍然不能独立把流程执行正确,还得安排人监督。更换模型则一次到位、且已实测验证(实测换用 Claude Opus 4.8 后,整套 skill 顺利跑通)→ 投入产出比(ROI)最高,因此定为最终方案。
与最初结论的衔接:复盘最初说"更换模型消除不了『工具能被自己改』这个根本原因,要和权限限制一起做、不能替代"——这是在"仍然继续用这台模型"的前提下成立的。实测之后的决定是直接更换这台模型:新模型本来就按流程执行、也不会擅自改动工具,"工具能被改"这个隐患根本不会触发,于是权限限制便从"必须做"降为"可选的补充防线"。
