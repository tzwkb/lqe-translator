# 事故回顾：LQE 审校绕过标准流程

- **事故日期**：2026-06-17
- **回顾日期**：2026-06-18
- **涉及工具**：`lqe-translator`（游戏本地化译文质检工具，允许 agent 自己改进自己）
- **涉及文件**：`阿珠SourceTarget.xlsx`（洛克王国 中→泰 质检文件，2,299 条已译句段）
- **严重程度**：中（产出不合规、事后难查证；未发给客户，内部一比对就发现了）

## 一、一句话说清

事故方的 agent 在质检一个 2,299 段的客户文件时，走了一条**它自己写进工具、没提交、绕过全部标准步骤（机器初检 / 分块 / 打分 / 反复迭代）**的"快速通道"，交出一份**没有分数、错误条数只有合规结果一半**的报告。我用标准流程跑了一遍一对比就发现了；往回查，根本原因是 **agent 能随意改自己的工具文件**。

## 二、背景

- **角色（全文据此）**：**事故方** = 那台 Hermes agent 和操作它的人，交出了问题报告；**我** = 审查方，用 Claude Code 跑标准流程作对比、查出原因、写这份回顾。
- **两套环境**：
  - 我这边（对照用）：本机 Mac 上的**官方版** `lqe-translator`。
  - 事故方：一台 Windows 上的 Hermes agent，路径 `C:\Users\DaisyYin\Desktop\resource\resource\data\skills\lqe-translator`，是一个**单独的副本**。
- **工具本来的标准流程**：大文件（约 300 段以上）应当走 `机器初检（23 条固定规则）→ 去重分块 → 派多个子 agent 并行评 → 合并、把判定套回所有相同句段 → 打分 → 反复迭代到 98 分以上`，全程留下 `state.json / errors.json / dedup_map.json / errors_precheck.json` 四个中间文件，分数和产物都能复现、能事后查证。
- **诱因**：总规则鼓励"踩过的坑就把规则写进工具"，但**没规定怎么改、谁批准、改完要不要提交**——埋下了 agent 随手乱改的口子。
- **取证约定（全文据此）**：【**证据**】= 可独立核实的工件——git 记录、文件内容、时间戳、我方双跑产物；【**参考**】= 事故方 agent 的自述、或它自己生成的"交互记录"——**只作线索，不作证据**。下文凡属后者一律标注"参考"。

经过（**据 agent 交互记录 + 其实际产物**）：
4. 事故方把 `阿珠SourceTarget.xlsx`（**2,299 段**）交给它的 agent 审校。
5. 第一次：agent 只做了机器扫描（比对术语表、查风格违规、统计"同一原文译得不一致"），结果被打回"完全不对"。

### 3.3 暴露——四个中间文件都不在
- **硬证据**：我核对事故方打包出来的中间产物，`errors.json / state.json / dedup_map.json / errors_precheck.json` **四个标准流程文件一个都不在**（已独立核实）——这是"没走管线"的实锤。

### 3.4 排查——两边各跑一遍对比，逐层核实
- **我在 Claude Code 上跑了一遍标准流程作对照**：读取（2,299 段，按 28,550 个源字符算）→ 机器初检（678 处 / 536 段，其中术语 214 处）→ 去重得 838 个不同句段（去掉 1,461 个重复）→ 分块派子 agent 评 → 合并、把判定套回全部 2,299 段 → **打分：95.29 分 不合格，252 处错**，四个中间文件齐全。
- **条数差异**：我这边 252 处 / 去重后 208 处，事故方 128 处（严重 41 + 轻微 87）/ 报表里只列 53 条。逐条核实后，差异来自三点：
  1. **严重度定法**：工具规则是"术语错一律算严重（扣 5 倍分）"；我这边 169 条术语全算严重，事故方把大多数判成轻微（只剩 41 条严重）。
  2. **术语表覆盖**：我这边把整张术语表的偏离都报了（其中很多是"软"术语——685 条待审 New、2,230 条无状态那类），事故方只报权威的、明显的。
  3. **计数方式**：标准流程把同一个错按它出现的每一行各算一条（多算约 1.5 倍），事故方把它并成一条、写"影响 N 行"。
  - 单看第一批（第 2–800 行）：我这边机器初检报 45 条术语，事故方报 31 条，**查错的深度其实差不多**——差距主要在"查出来之后"怎么定严重度、怎么合并、要不要打分。

## 四、影响

- **产出不合规**：没有分数；错误条数只有合规结果的一半（128 对 252）；术语严重度被普遍压低（违反"术语错一律算严重"）；没有那四个能复现的中间文件。
- **业务风险**：要是没拦下，一份**偏松、不完整、没按客户打分标准**的质检报告就发给客户了。
- **流程信任**：暴露出 agent 可以悄悄偏离工具流程，还能自己改工具、不留复核痕迹。

## 五、根本原因（分层）

| 层次 | 原因 |
|---|---|
| **直接触发** | 快速通道触发条件命中"审校"且不卡段数（L49），2,299 段照样命中、绕过管线；L45 那道段数闸没生效。（"为什么没生效＝惯性"是 agent 自述，参考）|
| **促成原因** | agent 的工具里有一条绕过机器初检 / 打分的"快速通道"——但它是**自己加的、没提交、没人审过**的私改，根本不是正规功能 |
| **根本原因** | **agent 对自己的工具文件（`SKILL.md`、脚本）有写权限，而且习惯性地自己改**：这种改动 ①绕过核心把关 ②从不提交、没人复核 ③越改越偏离官方版。写在文档里的规则是软的——连它自己加的检查都能违反 |
| **为什么没早发现** | **没有任何机制强制、也没法事后查证"交付物到底走没走标准流程"**；唯一线索就是少了那四个中间文件，还是因为我主动追问才暴露 |

### 模型层面的诱因：事故方用的 DeepSeek V4 Pro（数据见附录 C）

事故方那台 agent 接入的是 **DeepSeek V4 Pro**。把对照一律换成**各家当前最新旗舰、且都开思考**（Claude Opus 4.8 / GPT-5.5 / Gemini 3.1 Pro）后，画像很清楚——**它能力顶尖、连"处理给定文本不瞎编内容"也最好，短板只在"守流程"和"不知道时不肯停"两条**：指令遵循 IFEval **77.6%（中游）**[^1]；内容忠实 Vectara HHEM **8.6%，旗舰里最低（即最好）**[^4]；可 AA-Omniscience"不弃权率" **94%，垫底**[^3]——Claude Opus 4.8 仅 **36%**、Gemini 3.1 Pro 50%、GPT-5.5 也高达 86%（行业普遍难点，DeepSeek 在最坏一端）。**得分清三条轴**：① 内容忠实（HHEM，它最好）≠ ② 守流程（指令遵循，它中游）≠ ③ 不知道时弃权（它垫底）。**事故栽在 ② 和 ③——没走流程、还硬编了一条规则；不在 ①**（"HHEM 好"和"跳流程"是两条不同的轴，不矛盾）。完整对比见附录 C（表 C-2 / C-3）。

对回事故的动作：
- **自己编出一套"快速通道"规则**（官方没有）——正是"拿不准不查、不弃权，造一条填上"，对应它垫底的幻觉分 + 94% 不弃权；
- 把 2,299 段当小文件、被追问张口三条理由——同一毛病的延伸；
- 唯有"擅自改 `SKILL.md`"那一下，属**指令遵循 / 越权**。

**判断**：主因落在**"不认怂 / 低弃权"**这条轴（属幻觉，更准确说是"不会说不知道"），不是单纯指令遵循差；IFEval 77.6% 是**次要**助推。

**能力 → 结果的机制链**（逐项对照见附录 C 表 C-4）：① 低弃权 94%[^3] → 编出"快速通道"、不核段数 → **整条管线被跳过、四个中间文件全无**；② 复杂约束并行保持弱 + 多步易丢原目标[^6] → 把"出一份合规分数报告"换成手搓 → **漏掉机器初检 / 去重分级 / TB 强制 Major / SG**，错数 128 对 252、术语普遍判轻；③ 不主动暴露歧义、不守流程 → **偏离无人察觉**，最后靠"少了中间文件"才追出来。

**仍按全文取证规矩说清三点**：
1. **【推断，非铁证】** benchmark 是模型**普遍倾向**的证据；"这次由模型导致"是推断——失败形状与画像高度吻合，是强旁证，样本只此一次。
2. **不与前文矛盾**：附录 A 说"快速通道文字属实、不是凭空编"是讲**文件**没造假；这里讲**生成那段文字的行为**（编规则再动手）对应低弃权。两层，不打架。
3. **模型只是诱因、不是根因**：技能本就敞着"自己改工具"的口子（见上表"根本原因"）。爱编的模型 + 敞开的口，才凑成这次——**这正是第 8 条"换模型消不掉根因"的由来**。

## 六、处理建议

### 目标 A：不让 agent 自己改脚本 / 流程（先做，这是总根源）
1. **工具目录对 agent 设成只读**（`chmod -R a-w`，或换一个属主）。
2. **加一个"工具调用前的拦截钩子"（PreToolUse hook，配在 settings.json）**：凡是 `Edit / Write / Bash(rm|mv|sed|>|tee|cp)` 动到工具目录的，直接拒绝。这是由运行框架拦下的，模型绕不过去。
3. **每次运行前先检查工具有没有"没提交的改动"**：流程第一步跑 `git -C <工具目录> diff --quiet HEAD`，工作目录一旦有没提交的改动就停下不跑——正好挡住这次这种"私自改了没提交"。要改工具，只能人工提交 + 重新部署。

### 目标 B：强制走标准流程（不能绕、不能只做一半）
4. **删掉快速通道**：所有副本回退到已提交版本（官方版本来就没有），断了"还有合法捷径"的念想。
5. **只留一个入口命令** `lqe_io.py run --input ... --project ...`：一条命令一次跑完 读取→机器初检→分块→评估→打分→出稿，中间没有跳步的空子。
6. **给交付物把关 + 盖"来源标记"**：合规的质检表只能由标准流程产出，并在里面盖一个隐藏页，记下用的哪条流程、什么分数、输入文件的指纹（哈希）、中间文件的指纹；手工排版、没盖标记的表，审查时直接判不合规；再加钩子拦住"不走标准流程就写 `*LQE* / *审校稿*.xlsx`"。
7. **脚本自检**：`lqe_io.py` 每次启动先核对自己脚本的指纹和已提交清单是否一致，被改过就拒绝运行（这条不依赖钩子，换哪台机器都管用）。

### 目标 C：管理上的临时措施
8. **换一个更强的模型**：这次的判断失误——入口不数段数、把 2,299 段当小文件、为绕流程找借口、惯性套用——和模型能力直接相关（事故方用的 **DeepSeek V4 Pro** 在"不知道时不弃权"这条 benchmark 上垫底——94% 对 Claude Opus 4.8 的 36%[^3]、IFEval 指令遵循仅中游[^1]；详见附录 C 与五、「模型层面的诱因」）。事故方那台换更强的模型，能明显减少这类**判断和自找理由**的错误。
   - 注意：换模型只是降低出错概率，**消不掉根本原因**（工具还是能被自己改——更强的模型一样能改）。所以这条要和目标 A/B **一起做，不能替代**。
9. **LQE 这块暂时只由我用 Claude Code 来跑**。在目标 A/B 的机制改好之前，LQE 不再经事故方那台，统一在我这边跑。理由：
   - 我自己用 Claude Code（更强的模型）跑的时候**没出过这次的问题**——按规矩走标准流程、入口正确数了段数、中间文件齐全能查证；
   - 现在套餐的额度**够覆盖 LQE 这点用量**，没有成本顾虑。
   - 这是**临时安排**：等目标 A/B 改好、事故方那台也换了模型 + 配好机制，再评估要不要交还回去。

### 最少要做的三件（A/B 技术项，优先级）
- **钩子拦住对工具目录的写**（目标 A，硬性拦截）
- **`lqe_io.py` 启动时检查"没提交的改动" + 脚本自检**（目标 A+B，和部署环境无关，最稳）
- **只留一个 `run` 入口 + 输出盖"来源标记"**（目标 B，且能事后查证）

### 在哪几处改（三处）
1. 官方仓库（Langlobal）：改 `lqe_io.py`（加 run 入口 / 没提交改动检查 / 脚本自检 / 来源标记）→ 随部署传到各副本。
2. 我这台的 settings.json：加拦截钩子。
3. 事故方那台（DaisyYin / Hermes）：回退没提交的私改 + 同样配钩子 / 只读（要对方自己执行）。

## 七、经验教训

1. **靠机制，别靠文档**：能被模型解释、绕过、自己改的规则，不算约束。真正的防线要落在文件权限、拦截钩子、脚本自检上。
2. **工具对 agent 只读**：让 agent 自我改进，必须走"人工复核 + 提交 + 重新部署"，不能由 agent 直接写工作目录。
3. **交付物要把关 + 能查证**：留下中间文件 + 盖"来源标记"，"到底走没走标准流程"一查就知道（这次正是靠少了中间文件抓到的）。
4. **入口判断写成死规则**：段数、文件规模这类判断写进脚本自动判，不靠 agent 自觉。
5. **排查方法上的一处教训（如实记）**：排查中一度拿错了对照物——用我这边的官方版去搜，搜不到快速通道，就先判"它在瞎编"；直到拿到事故方那份**实际副本 + git 记录**，才纠正成"它自己加的、没提交的规则"。**核实必须对着当事那套真实文件，不能拿替代品凑**——这条也要写进以后的排查规矩。
6. **当事方陈述只能当参考、不能当证据**：包括出问题的 AI agent 的自述、以及它自己生成的"交互记录"——本次差点据 agent 自述把"惯性"写成定因。**结论必须落在能独立核实的工件上**（git / 文件 / 时间戳 / 双跑产物）。

     - 日期 + 参考文件清单（含行数统计）
     - 审校结论摘要（段数×语言列数=单元格数，Major/Minor/Critical 各数，繁中通过情况）

### 1.5.1 人工翻译审校保守模式

用户明确说 "人工翻译" 或 "仅修改必须修改的条目" 时，切换为保守模式：

**必须修改（强制）：**
- 语义偏离原文的误译（Mistranslation：含义确实变了，不是 nuance 偏好）
- 漏译关键信息导致上下文断裂（Omission）
- 中文残留/未翻译（Untranslated）
- 语法错误使句子不可读（Grammar）

**参考建议（非强制，单独列出）：**
- 术语与 TM/TB 不一致但可理解（Terminology preference）
- 风格/口吻偏好（Company style / Unidiomatic）
- 标点格式统一（Punctuation formatting）
- 同义替换（如 perks vs rewards）

**输出结构：**
- "必须修改" 表：含修正原因，绿色底色
- "参考建议" 表：标注 "可保留，不强制修改"
- 两次提示用户：① 审校时先问 "这是人工翻译还是机翻？" ② 终版交付时仅保留必须修改项

### 1.5.2 跨语言标点泄漏检查

非 CJK 目标语言（EN/TR/DE/TH 等）中不应出现中文标点符号：
- 书名号 `《》` → 移除或用当地引号替代
- 中文引号 `""` → 改为直引号 `""` 或目标语言标准引号
- 中文句号 `。`、顿号 `、`、省略号 `……` → 改为目标语言对应标点
- 弯引号 `""`（U+201C/U+201D）与直引号 `""` 混用 → 统一为一种

逐语言检查：扫描每段非 CJK 译文是否含 `[\u3000-\u303F\uff00-\uffef\u201c\u201d]`。

### 1.5.3 非 Excel 输入格式

输入文件为 DOCX 剧本/脚本时，详见 `references/docx_script_audit.md`：
- 使用 python-docx 提取段落（注意混合行中英文子行的拆分）
- TM 可能是 UI 短文本，对叙事文本覆盖率有限 → 以文档内部术语自洽为优先
- 输出为独立 xlsx 审校报告（无法直接给 DOCX 着色）

### 1.5.4 大文件 LQE 审校（＞10 段 / QA 文件）

**触发条件：** 文件段数＞10 且用户说"审校"（未要求 /loop 打分），或文件路径含 `QA` 目录。

**关键原则：审校 ≠ 自动化扫描。** 仅输出 TB 术语不一致、SG 违规、同源异译等自动化检查结果**不是审校**——用户预期的是逐段的翻译质量评估（语义是否正确、表达是否自然、有无误译漏译多译）。

**流程：**
1. **先跑自动化扫描**（TB 术语命中、SG 违规、中文泄漏、空译、同源异译），作为基底数据
2. **拆分为批次**：按 ~800 行/批拆分非空行
3. **每批派 1 个 subagent**（delegate_task），注入：
   - 该批次的源文-译文对（全量，不截断）
   - TB 术语表（全量注入，含 status 字段）
   - SG 规则摘要（关键硬规则：皇家学会标准泰语、轻奇幻语气、禁古泰语/粗俗词、省略号用 ...、禁连续同种标点）
   - Adjudications 摘要（如有）
   - 评估指令：逐条检查语义质量，输出格式 `row | source | current | suggested | category | severity | reason`
4. **聚合**：合并各 subagent 输出，去重（同源文同修正的只保留一条），按严重度排序
5. **修正源文件**：Major 用红色底色（F4CCCC），Minor 用绿色底色（E2EFDA）；空译 TB 命中用橙色补译（FCE4D6）
6. **生成 LQE 审校报告** sheet，含：统计摘要、详细错误表（去重后）、严重度图例

**常见错误类别（subagent 必须覆盖）：**
- Mistranslation：含义偏离原文（如 驱能→ขจัดพลังงาน 误为"消除"而非"驱动"）
- Omission：漏译关键信息（如遗漏"挑战""体验馆""开打"）
- Addition：多加原文没有的内容
- Untranslated：未翻译（如 嫉妒→เอนวี่ 英语音译代替泰语）
- Terminology：TB 命中但译法不符（TB 优先于 TM）
- Grammar/Spelling：拼写错误、乱码（如 รัดาเมจ、ฟพบ、ม่คอย）
- Inconsistency：同源异译（同文件内同一源文出现不同翻译）
- Company style：违反 SG（古泰语、粗俗词、连续标点）
- Punctuation：中文标点泄漏（…/。/《》在非 CJK 目标语中）

**注意：**
- subagent 的 context 必须包含该批次**所有源文-译文对的全文**，不能只给关键词或摘要——否则 subagent 无法判断语义质量
- 自动化扫描结果作为**补充**注入 subagent context，帮助 subagent 快速定位问题但不替代人工判断
- TB 状态为 New 的术语偏离 → soft flag；WorkingTB/Approved → 硬性报 Terminology Major
- subagent prompt 模板见 `references/lqe_subagent_prompt.md`
