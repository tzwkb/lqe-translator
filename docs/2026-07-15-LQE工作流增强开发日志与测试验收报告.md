## 4. 任务 B 开发记录：SDLXLIFF 原生导入

### 4.1 输入发现与解析

新增：

```text
scripts/lqe_inputs/__init__.py
scripts/lqe_inputs/sdlxliff.py
```

支持内容：

- 单个 `.sdlxliff` 文件。
- 递归读取 SDLXLIFF 目录并按相对路径稳定排序。
- XLIFF 1.2 多 `<file>`、多 TU 和多分段内容。
- `source`、`seg-source`、`target`、`mrk mid` 和 `sdl:seg-defs` 配对。
- XLIFF 内联标签、尾文本、空白和原始 XML 证据。
- SDL confirmation、origin、percent、text-match、locked、comment 和修改人信息。
- TU、Segment 和缺失业务 ID 的稳定 `source_ref`。

分段源文存在但 `<target/>` 为空时，每个源分段保留为空译文，不再因缺少 target marker 失败。

### 4.2 扩展与 XML 安全边界

未知厂商扩展按以下规则处理：

- 不影响句段边界时，保留 XML、属性和 namespace 证据。
- 厂商扩展节点隐藏 `file`、`body`、`trans-unit`、`source`、`target` 或 `seg-defs` 时失败。
- 本应唯一的直接 `body`、`source`、`seg-source`、`target` 或 `sdl:seg-defs` 重复时失败；多个直接 `<file>` 和多个直接 `<trans-unit>` 仍受支持。
- 影响 `mid` 或 source/target 配对时失败，不猜测结构。
- namespace 前缀重绑定时，显示映射保持首次绑定，manifest 记录全部声明 URI。

每个输入文件只读取一次。DTD/entity 预检、ElementTree 解析和 SHA-256 均使用同一份不可变字节快照，避免校验内容与实际解析内容不一致。

### 4.3 Manifest、语言和规则

每个 SDLXLIFF job 生成：

```text
source_manifest.json
tm_candidates.json
scope.json
state.json
```

`source_manifest.json` 记录：

- 输入相对路径和 SHA-256。
- 语言声明。
- namespace 与扩展证据。
- content type 和 exclusion 规则命中。
- 纳入、排除和保护证据。

同批文件的语言声明必须一致。区域语言可回退到基础语言资源，例如：

```text
en-US → en
```

内容类型和排除只来自 profile 的显式规则，不根据 CC、FF、文件名或目录名推断。

### 4.4 TM 与保护决策

默认策略为：

```text
candidate-only
```

只有同时满足以下三项的 Segment 才进入 `tm_candidates.json`：

```text
origin = tm
percent = 100
text-match = SourceAndTarget
```

候选不会自动免检或免计分。保护方式有两种：

1. 使用 `protect-segments` 显式确认候选。
2. 在 profile 或 CLI 显式启用严格自动保护策略。

`locked` 段始终保护。TM 候选文件绑定当前 job 路径、`source_ref` 和原始证据；跨 job、篡改候选或直接把 candidate 文件传给计分器均被拒绝。

### 4.5 报告与 corrected 输出

SDLXLIFF 的 `LQE Results` 固定为 11 列：

```text
来源文件
TU ID
SDL Segment ID
原文
原译
建议译文
处理方式
错误详情
LQE_Iter
Protected
Protection Evidence
```

corrected Excel 固定为 5 列：

```text
来源文件
TU ID
SDL Segment ID
原文
译文
```

第一版不回写 SDLXLIFF XML。原始 XML 保持不变，交付仍为标准 corrected Excel。

## 5. 任务 B 开发记录：无术语检查模式

### 5.1 生效方式

任务明确要求不检查术语时，初始化命令使用：

```bash
--no-terminology
```

该参数高于 profile 中的术语配置，且不能与显式 `--terminology` 同时使用。最终检查范围同时写入：

```text
state.check_scope
scope.json
```

### 5.2 模块范围

标准模式继续使用：

```text
terminology
accuracy
grammar
naturalness
```

无术语模式使用：

```text
precheck_review
accuracy
grammar
naturalness
```

无术语模式关闭：

```text
terminology
proper_names
term_audit
```

仍保留 Markup、数字、长度、Locale convention、Company style、文件内一致性和其他非术语预检。

### 5.3 范围与归属防线

实现以下校验：

- 禁止显式加载或残留读取术语文件。
- 禁止 Terminology 类别、`TERM REVIEW:` 和 `confirmed_term` 证据进入结果。
- 校验模块、类别、严重度、comment 和 issue schema。
- `precheck_review` 只能引用当前 Segment 的原始非术语预检问题。
- 不同 Segment 的 precheck 不参与跨 ID 错误去重。
- split、batch、merge、calc、apply、write、export 和 finalize 均重新校验 scope。

标准模式的原有四模块行为保持兼容，不因无术语模式引入而改变。

## 6. 终检阶段发现并修复的问题

终检不是重复运行同一测试。审查过程中发现了以下可复现问题，并在最终 Head 前修复：

| 问题 | 风险 | 修复结果 |
|---|---|---|
| DTD 预检、解析和哈希读取不同文件状态 | TOCTOU，证据与解析不一致 | 改为单一字节快照 |
| 重复或隐藏 XML 结构被忽略 | 静默漏段、locked 证据丢失 | 重复/隐藏结构明确失败 |
| 空分段 target 被当作结构错误 | 合法空译文无法导入 | 按源 mid 生成空译文 |
| namespace 重绑定只记录首次 URI | 扩展证据不完整 | 记录全部声明 |
| Job 正式产物直接覆盖或回滚竞争者文件 | 并发数据损坏 | 原子 no-clobber 发布及 inode 所有权回滚 |
| 无术语检查可被大小写、未知 category/severity 或空 comment 绕过 | 禁用范围失效 | 所有入口统一 schema 和 scope 校验 |
| 不同 Segment 的 precheck 被去重或复制 | 问题丢失或错配 | dedup key 纳入当前 ID 的 precheck 证据 |
| TM candidate 可跨 job、被篡改或直接免计分 | 未确认段被保护 | 候选归属校验，必须形成保护决定 |
| `protect-segments` 部分发布失败 | state 与保护清单不一致 | 事务发布，state 最后作为提交标记 |
| 零词数但存在计分错误仍可能 PASS | 错误通过 | 有惩罚时返回 FAIL |
| Excel 外部文本被识别为公式 | 公式注入或显示异常 | 外部内容强制写为文本 |
| finalizer 子步骤失败未完整传播 | 生成不完整交付 | 失败即停止且不写 finalized 标记 |
