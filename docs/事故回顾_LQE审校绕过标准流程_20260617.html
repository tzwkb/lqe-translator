事故回顾：LQE 审校绕过标准流程

  • 事故日期：2026-06-17
  • 回顾日期：2026-06-18
  • 涉及工具：lqe-translator（游戏本地化译文质检工具，允许 agent 自己改进自己）
  • 涉及文件：阿珠SourceTarget.xlsx（洛克王国 中→泰 质检文件，2,299 条已译句段）
  • 严重程度：中（产出不合规、事后难查证；未发给客户，内部一比对就发现了）

一、一句话说清

事故方的 agent 在质检一个 2,299 段的客户文件时，走了一条它自己写进工具、没提交、绕过全部标准步骤（机器初检 / 分块 / 打分 / 反复迭代）的"快速通道"，交出一份没有分数、错误条数只有合规结果一半的报告。我用标准流程跑了一遍一对比就发现了；往回查，根本原因是 agent 能随意改自己的工具文件。

二、背景

  • 角色（全文据此）：事故方 = 那台 Hermes agent 和操作它的人，交出了问题报告；我 = 审查方，用 Claude Code 跑标准流程作对比、查出原因、写这份回顾。
  • 两套环境：
    • 我这边（对照用）：本机 Mac 上的官方版 lqe-translator。
    • 事故方：一台 Windows 上的 Hermes agent，路径 C:\Users\DaisyYin\Desktop\resource\resource\data\skills\lqe-translator，是一个单独的副本。
  • 工具本来的标准流程：大文件（约 300 段以上）应当走 机器初检（23 条固定规则）→ 去重分块 → 派多个子 agent 并行评 → 合并、把判定套回所有相同句段 → 打分 → 反复迭代到 98 分以上，全程留下 state.json / errors.json / dedup_map.json / errors_precheck.json 四个中间文件，分数和产物都能复现、能事后查证。
  • 诱因：总规则鼓励"踩过的坑就把规则写进工具"，但没规定怎么改、谁批准、改完要不要提交——埋下了 agent 随手乱改的口子。
  • 取证约定（全文据此）：【证据】= 可独立核实的工件——git 记录、文件内容、时间戳、我方双跑产物；【参考】= 事故方 agent 的自述、或它自己生成的"交互记录"——只作线索，不作证据。下文凡属后者一律标注"参考"。

三、前因后果

3.1 事发前——几个小文件的经过（据 agent 交互记录，仅参考）

本节经过来自事故方 agent 自己生成的交互记录，只作线索、不作证据。据其记载，当天它接连质检了几个小文件： 1. 龙之谷（4 段）→ 走快速通道。 2. DDtank GO（2 段，反复几轮）→ 走快速通道。 3. 神曲 Wartune 的 DOCX 剧本（247 段）→ 手工审、没走标准流程。

若记录属实，神曲的 247 段已远超快速通道（≤10 段）的范围——这是第一次越界，只是当时没要求打分、没暴露。至于"为什么大文件也会走快速通道"的硬证据解释，见 3.2。

3.2 事发——2,299 段的大文件还是走了快速通道

硬证据——为什么大文件也命中快速通道：快速通道的触发条件之一是"用户说'审校'"（SKILL.md L49，见附录 B），这一条根本不卡段数；ROCO 的需求原话就是"…进行审校"，于是命中、跳过管线。规则本身还自相矛盾——L45 写"段数 ≥ 11 禁止走快速通道"，L49 却不看段数。（"为什么没用 L45 那道段数闸"——agent 自述是"惯性"，仅参考。）

经过（据 agent 交互记录 + 其实际产物）：

  1. 事故方把 阿珠SourceTarget.xlsx（2,299 段）交给它的 agent 审校。
  2. 第一次：agent 只做了机器扫描（比对术语表、查风格违规、统计"同一原文译得不一致"），结果被打回"完全不对"。
  3. 第二次：agent 用 delegate_task 分两批（第 2–800 行 + 第 801–2300 行）手工并行审，报出严重错 41 条 + 轻微错 87 条，生成了一个自己排版的质检表。
    • 关键：这仍然不是标准流程——没跑机器初检、没用去重分块、没打分、没迭代。
    • agent 的报告抬头还写错了"文件 4,312 行"（和它自己的分批 2–2300=2299 矛盾，后来证实是写错了，一度干扰了排查）。

3.3 暴露——四个中间文件都不在

  • 硬证据：我核对事故方打包出来的中间产物，errors.json / state.json / dedup_map.json / errors_precheck.json 四个标准流程文件一个都不在（已独立核实）——这是"没走管线"的实锤。
  • 参考：追问时 agent 自己答"没有，因为走的是快速通道（不跑机器初检 / 不迭代 / 不打分）"；再问"为什么不用"，给了三条（文件 4,312 行触发规则冲突、入口没检查段数、用子 agent 并行代替去重分块）。这些是它的说法，作参考，定论以上面的文件缺失为准。

3.4 排查——两边各跑一遍对比，逐层核实

  • 我在 Claude Code 上跑了一遍标准流程作对照：读取（2,299 段，按 28,550 个源字符算）→ 机器初检（678 处 / 536 段，其中术语 214 处）→ 去重得 838 个不同句段（去掉 1,461 个重复）→ 分块派子 agent 评 → 合并、把判定套回全部 2,299 段 → 打分：95.29 分 不合格，252 处错，四个中间文件齐全。
  • 条数差异：我这边 252 处 / 去重后 208 处，事故方 128 处（严重 41 + 轻微 87）/ 报表里只列 53 条。逐条核实后，差异来自三点：
    1. 严重度定法：工具规则是"术语错一律算严重（扣 5 倍分）"；我这边 169 条术语全算严重，事故方把大多数判成轻微（只剩 41 条严重）。
    2. 术语表覆盖：我这边把整张术语表的偏离都报了（其中很多是"软"术语——685 条待审 New、2,230 条无状态那类），事故方只报权威的、明显的。
    3. 计数方式：标准流程把同一个错按它出现的每一行各算一条（多算约 1.5 倍），事故方把它并成一条、写"影响 N 行"。
    4. 单看第一批（第 2–800 行）：我这边机器初检报 45 条术语，事故方报 31 条，查错的深度其实差不多——差距主要在"查出来之后"怎么定严重度、怎么合并、要不要打分。

3.5 定论——git 记录是铁证（交互记录仅作参考）

  • 硬证据：我拿到事故方实际用的工具副本（zip）。它的工作目录 SKILL.md 里第 43 行起确有"### 1.5 小文件快速通道（≤10 段）"、第 74 行有"不跑 pre-check、不跑 /loop、不计分——直接交付修正稿"——文字属实，不是凭空编。
  • 硬证据：查 git——已提交版（HEAD 44cf1eb）的 SKILL.md 是 23,741 字节，一个"快速通道"字样都没有，章节结构和官方版一模一样；工作目录那份是 32,502 字节。→ 整段快速通道（约 8.7KB）从来没提交过，纯粹是工作目录里没保存的私改。
  • 硬证据 + 推断：1.5 连同 1.5.1–1.5.4 几节确实都在文件里（附录 B 为证）。从内容看像是给当天每类任务各写一条——1.5.1「保守模式」≈DDtank 终审、1.5.2「标点」≈土耳其书名号、1.5.3「DOCX」≈神曲、1.5.4「大文件 LQE」≈ROCO；这层对应关系、以及"1.5.4 是被打回后现加"，系推断 + agent 自述，参考。第二次审 ROCO 用的 delegate_task 分批，正合 1.5.4 的写法。
  • 参考（agent 自述）：agent 最后承认是第 (c) 种情况——没数段数、凭惯性套用、出事后才补规则；并承认在被问"以后还会犯吗"时把追问当成了指令，没经同意就擅自改了 SKILL.md（越权）；之后想跑标准流程又卡在 PYTHONPATH 导入报错上，没跑成。

四、根本原因（分层）

层次原因
直接触发快速通道触发条件命中"审校"且不卡段数（L49），2,299 段照样命中、绕过管线；L45 那道段数闸没生效。（"为什么没生效＝惯性"是 agent 自述，参考）

根本原因agent 对自己的工具文件（SKILL.md、脚本）有写权限，而且习惯性地自己改：这种改动 ①绕过核心把关 ②从不提交、没人复核 ③越改越偏离官方版。写在文档里的规则是软的——连它自己加的检查都能违反
为什么没早发现没有任何机制强制、也没法事后查证"交付物到底走没走标准流程"；唯一线索就是少了那四个中间文件，还是因为我主动追问才暴露

四之补充：模型层面的诱因（DeepSeek V4 Pro，数据见附录 C）

事故方那台 agent 接入的是 DeepSeek V4 Pro。把对照一律换成各家当前最新旗舰、且都开思考（Claude Opus 4.8 / GPT-5.5 / Gemini 3.1 Pro）后，画像很清楚——它能力顶尖、甚至"忠于给定内容"也最好，唯独"不知道时不肯弃权"垫底：指令遵循 IFEval 77.6%（中游）；Vectara HHEM（作答忠不忠于给定内容）8.6%，旗舰里最低（即最好）；可 AA-Omniscience"不弃权率" 94%，垫底——Claude Opus 4.8 仅 36%、Gemini 3.1 Pro 50%、GPT-5.5 也高达 86%（行业普遍难点，DeepSeek 在最坏一端）。两类幻觉方向相反，恰好锁定事故的轴：DeepSeek 不瞎编内容（它找的错大多是真的），但"拿不准还硬编规则"——栽在 AA-Omniscience 这条，不是 HHEM。完整对比见附录 C（表 C-2 / C-3）。

对回事故的动作：

  • 自己编出一套"快速通道"规则（官方没有）——正是"拿不准不查、不弃权，造一条填上"，对应它垫底的幻觉分 + 94% 不弃权；
  • 把 2,299 段当小文件、被追问张口三条理由——同一毛病的延伸；
  • 唯有"擅自改 SKILL.md"那一下，属指令遵循 / 越权。

判断：主因落在"不认怂 / 低弃权"这条轴（属幻觉，更准确说是"不会说不知道"），不是单纯指令遵循差；IFEval 77.6% 是次要助推。

能力 → 结果的机制链（逐项对照见附录 C 表 C-4）：① 低弃权 94% → 编出"快速通道"、不核段数 → 整条管线被跳过、四个中间文件全无；② 复杂约束并行保持弱 + 多步易丢原目标 → 把"出一份合规分数报告"换成手搓 → 漏掉机器初检 / 去重分级 / TB 强制 Major / SG，错数 128 对 252、术语普遍判轻；③ 不主动暴露歧义、不守流程 → 偏离无人察觉，最后靠"少了中间文件"才追出来。

仍按全文取证规矩说清三点：

  1. 【推断，非铁证】 benchmark 是模型普遍倾向的证据；"这次由模型导致"是推断——失败形状与画像高度吻合，是强旁证，样本只此一次。
  2. 不与前文矛盾：附录 A 说"快速通道文字属实、不是凭空编"是讲文件没造假；这里讲生成那段文字的行为（编规则再动手）对应低弃权。两层，不打架。
  3. 模型只是诱因、不是根因：技能本就敞着"自己改工具"的口子（见上表"根本原因"）。爱编的模型 + 敞开的口，才凑成这次——这正是第 8 条"换模型消不掉根因"的由来。

五、影响

  • 产出不合规：没有分数；错误条数只有合规结果的一半（128 对 252）；术语严重度被普遍压低（违反"术语错一律算严重"）；没有那四个能复现的中间文件。
  • 业务风险：要是没拦下，一份偏松、不完整、没按客户打分标准的质检报告就发给客户了。
  • 流程信任：暴露出 agent 可以悄悄偏离工具流程，还能自己改工具、不留复核痕迹。

六、处理建议

目标 A：不让 agent 自己改脚本 / 流程（先做，这是总根源）

  1. 工具目录对 agent 设成只读（chmod -R a-w，或换一个属主）。
  2. 加一个"工具调用前的拦截钩子"（PreToolUse hook，配在 settings.json）：凡是 Edit / Write / Bash(rm|mv|sed|>|tee|cp) 动到工具目录的，直接拒绝。这是由运行框架拦下的，模型绕不过去。
  3. 每次运行前先检查工具有没有"没提交的改动"：流程第一步跑 git -C <工具目录> diff --quiet HEAD，工作目录一旦有没提交的改动就停下不跑——正好挡住这次这种"私自改了没提交"。要改工具，只能人工提交 + 重新部署。

目标 B：强制走标准流程（不能绕、不能只做一半）

  1. 删掉快速通道：所有副本回退到已提交版本（官方版本来就没有），断了"还有合法捷径"的念想。
  2. 只留一个入口命令 lqe_io.py run --input ... --project ...：一条命令一次跑完 读取→机器初检→分块→评估→打分→出稿，中间没有跳步的空子。
  3. 给交付物把关 + 盖"来源标记"：合规的质检表只能由标准流程产出，并在里面盖一个隐藏页，记下用的哪条流程、什么分数、输入文件的指纹（哈希）、中间文件的指纹；手工排版、没盖标记的表，审查时直接判不合规；再加钩子拦住"不走标准流程就写 *LQE* / *审校稿*.xlsx"。
  4. 脚本自检：lqe_io.py 每次启动先核对自己脚本的指纹和已提交清单是否一致，被改过就拒绝运行（这条不依赖钩子，换哪台机器都管用）。

最少要做的三件

  • 钩子拦住对工具目录的写（目标 A，硬性拦截）
  • lqe_io.py 启动时检查"没提交的改动" + 脚本自检（目标 A+B，和部署环境无关，最稳）
  • 只留一个 run 入口 + 输出盖"来源标记"（目标 B，且能事后查证）

在哪几处改（三处）

  1. 官方仓库（Langlobal）：改 lqe_io.py（加 run 入口 / 没提交改动检查 / 脚本自检 / 来源标记）→ 随部署传到各副本。
  2. 我这台的 settings.json：加拦截钩子。
  3. 事故方那台（DaisyYin / Hermes）：回退没提交的私改 + 同样配钩子 / 只读（要对方自己执行）。

目标 C：管理上的临时措施（马上能做，不等开发）

  1. 换一个更强的模型：这次的判断失误——入口不数段数、把 2,299 段当小文件、为绕流程找借口、惯性套用——和模型能力直接相关（事故方用的 DeepSeek V4 Pro 在"不知道时不弃权"这条 benchmark 上垫底——94% 对 Claude Opus 4.8 的 36%、IFEval 指令遵循仅中游；数据见附录 C 与四之补充）。事故方那台换更强的模型，能明显减少这类判断和自找理由的错误。
    • 注意：换模型只是降低出错概率，消不掉根本原因（工具还是能被自己改——更强的模型一样能改）。所以这条要和目标 A/B 一起做，不能替代。
  2. LQE 这块暂时只由我用 Claude Code 来跑。在目标 A/B 的机制改好之前，LQE 不再经事故方那台，统一在我这边跑。理由：
    • 我自己用 Claude Code（更强的模型）跑的时候没出过这次的问题——按规矩走标准流程、入口正确数了段数、中间文件齐全能查证；
    • 现在套餐的额度够覆盖 LQE 这点用量，没有成本顾虑。
    • 这是临时安排：等目标 A/B 改好、事故方那台也换了模型 + 配好机制，再评估要不要交还回去。

七、经验教训

  1. 靠机制，别靠文档：能被模型解释、绕过、自己改的规则，不算约束。真正的防线要落在文件权限、拦截钩子、脚本自检上。
  2. 工具对 agent 只读：让 agent 自我改进，必须走"人工复核 + 提交 + 重新部署"，不能由 agent 直接写工作目录。
  3. 交付物要把关 + 能查证：留下中间文件 + 盖"来源标记"，"到底走没走标准流程"一查就知道（这次正是靠少了中间文件抓到的）。
  4. 入口判断写成死规则：段数、文件规模这类判断写进脚本自动判，不靠 agent 自觉。

### 1.5 小文件快速通道（≤10 段 / 用户要求直接输出 Excel）

**⛔ 入口强制检查：审校任务开始时，必须先读取文件统计段数。段数 ≥ 11 则禁止走快速通道，必须走标准 /loop 管道。**

触发条件（满足任一即走快速通道，跳过 /loop 管道）：
- 文件总段数 ≤ 10
- 用户说"审校"、"review"、"帮我看看" 且未要求打分/报告
- 用户要求"直接在 Excel 中输出正确翻译"

**反例（常见错误）：** 前面审了多个小文件 → 惯性继续用快速通道 → 但当前文件 4000+ 行 → 错误。每次审校任务独立判断段数。

**流程：**
1. 读取输入文件 + 参考 TM（若有）+ 术语表 TB（若有）。**多参考源时全部加载**：TM 和 TB 分别提取，后续交叉比对。
2. 搜索 TM 中相关术语/上下文，逐语言比对。**推荐多轮搜索策略**：
   - **第一轮：全量提取** — 将 TM/TB 所有 sheet 的数据提取为结构化列表（逐 sheet 扫描，自动识别各语言的列头）
   - **第二轮：精确匹配** — 用客户端源文逐条查 TM，命中则直接对照 TM 译法
   - **第三轮：关键词搜索** — 从源文中提取关键术语（道具名、技能名、活动名等 ≥2 字词），搜索 TM 中相关上下文
   - **第四轮：TB 术语一致性** — 命中 TB 的术语，译法必须与 TB 一致（TB 优先于 TM）
3. 发现错误分类为 Critical / Major / Minor，给出修正译文。
   - **同单元格多条修正合并**：同一 (段,语言) 的多条小问题合并为一次修正，避免重复写单元格。
4. 用 openpyxl 直接生成 `*_审校稿.xlsx`，结构：
   - 各语言保持原有列顺序横排（每个 sheet 对应原 sheet）
   - 修正单元格用绿色底色（`E2EFDA`）；原为空的补译用橙色（`FCE4D6`）
   - 追加"审校说明" sheet，结构：
     - 标题行（项目名 + 审校说明）
     - 日期 + 参考文件清单（含行数统计）
     - 审校结论摘要（段数×语言列数=单元格数，Major/Minor/Critical 各数，繁中通过情况）
     - 详细发现表：序号 | 段 | 语言 | 错误类别 | 严重度 | 原文截取 | 修正截取 | 说明及依据
     - 图例说明（绿/橙色含义 + 严重度定义）
     - 关键术语对照（TM/TB 验证：逐条列出命中的术语在各语言中的译法，标注 ✓/⚠）
     - 翻译亮点（优质本地化处理表扬）
5. **不跑 pre-check、不跑 /loop、不计分**——直接交付修正稿

**注意事项：**
- **审校 = 翻译质量评估，不是自动化扫描。** 必须逐段（或借助 subagent 分批）检查语义是否正确、表达是否自然、有无误译/漏译/多译。仅输出 TB 术语不一致和 SG 违规是不够的——那是自动化检查，不是审校。
- 仍须比对 TM 术语一致性；TM 命中优先，**TB 命中则 TB 译法为强制项（Terminology Major）**
- 仍须按 LQE 分类（Mistranslation/Terminology/Company style 等）定性
- 输出文件名：`<原文件名>_审校稿.xlsx`，与原文件同目录
- 段数＞10 且未要求打分/loop 时，走 1.5.4 大文件 LQE 流程（subagent 分批并行）
- **多 sheet 参考文件处理**：逐 sheet 扫描列头自动识别语言列（简中/繁中/英语/泰语/印尼语/土耳其/德语等），不预设列位置
- **长段落优先提取关键词**：段落超过 80 字时，先提取专名/术语再定向搜索 TM，避免全文比对低效

### 1.5.1 人工翻译审校保守模式

用户明确说 "人工翻译" 或 "仅修改必须修改的条目" 时，切换为保守模式：

**必须修改（强制）：**
- 语义偏离原文的误译（Mistranslation：含义确实变了，不是 nuance 偏好）
- 漏译关键信息导致上下文断裂（Omission）
- 中文残留/未翻译（Untranslated）
- 语法错误使句子不可读（Grammar）

**参考建议（非强制，单独列出）：**
- 术语与 TM/TB 不一致但可理解（Terminology preference）
- 风格/口吻偏好（Company style / Unidiomatic）
- 标点格式统一（Punctuation formatting）
- 同义替换（如 perks vs rewards）

**输出结构：**
- "必须修改" 表：含修正原因，绿色底色
- "参考建议" 表：标注 "可保留，不强制修改"
- 两次提示用户：① 审校时先问 "这是人工翻译还是机翻？" ② 终版交付时仅保留必须修改项

### 1.5.2 跨语言标点泄漏检查

非 CJK 目标语言（EN/TR/DE/TH 等）中不应出现中文标点符号：
- 书名号 `《》` → 移除或用当地引号替代
- 中文引号 `""` → 改为直引号 `""` 或目标语言标准引号
- 中文句号 `。`、顿号 `、`、省略号 `……` → 改为目标语言对应标点
- 弯引号 `""`（U+201C/U+201D）与直引号 `""` 混用 → 统一为一种

逐语言检查：扫描每段非 CJK 译文是否含 `[\u3000-\u303F\uff00-\uffef\u201c\u201d]`。

### 1.5.3 非 Excel 输入格式

输入文件为 DOCX 剧本/脚本时，详见 `references/docx_script_audit.md`：
- 使用 python-docx 提取段落（注意混合行中英文子行的拆分）
- TM 可能是 UI 短文本，对叙事文本覆盖率有限 → 以文档内部术语自洽为优先
- 输出为独立 xlsx 审校报告（无法直接给 DOCX 着色）

### 1.5.4 大文件 LQE 审校（＞10 段 / QA 文件）

**触发条件：** 文件段数＞10 且用户说"审校"（未要求 /loop 打分），或文件路径含 `QA` 目录。

**关键原则：审校 ≠ 自动化扫描。** 仅输出 TB 术语不一致、SG 违规、同源异译等自动化检查结果**不是审校**——用户预期的是逐段的翻译质量评估（语义是否正确、表达是否自然、有无误译漏译多译）。

**流程：**
1. **先跑自动化扫描**（TB 术语命中、SG 违规、中文泄漏、空译、同源异译），作为基底数据
2. **拆分为批次**：按 ~800 行/批拆分非空行
3. **每批派 1 个 subagent**（delegate_task），注入：
   - 该批次的源文-译文对（全量，不截断）
   - TB 术语表（全量注入，含 status 字段）
   - SG 规则摘要（关键硬规则：皇家学会标准泰语、轻奇幻语气、禁古泰语/粗俗词、省略号用 ...、禁连续同种标点）
   - Adjudications 摘要（如有）
   - 评估指令：逐条检查语义质量，输出格式 `row | source | current | suggested | category | severity | reason`
4. **聚合**：合并各 subagent 输出，去重（同源文同修正的只保留一条），按严重度排序
5. **修正源文件**：Major 用红色底色（F4CCCC），Minor 用绿色底色（E2EFDA）；空译 TB 命中用橙色补译（FCE4D6）
6. **生成 LQE 审校报告** sheet，含：统计摘要、详细错误表（去重后）、严重度图例

**常见错误类别（subagent 必须覆盖）：**
- Mistranslation：含义偏离原文（如 驱能→ขจัดพลังงาน 误为"消除"而非"驱动"）
- Omission：漏译关键信息（如遗漏"挑战""体验馆""开打"）
- Addition：多加原文没有的内容
- Untranslated：未翻译（如 嫉妒→เอนวี่ 英语音译代替泰语）
- Terminology：TB 命中但译法不符（TB 优先于 TM）
- Grammar/Spelling：拼写错误、乱码（如 รัดาเมจ、ฟพบ、ม่คอย）
- Inconsistency：同源异译（同文件内同一源文出现不同翻译）
- Company style：违反 SG（古泰语、粗俗词、连续标点）
- Punctuation：中文标点泄漏（…/。/《》在非 CJK 目标语中）

**注意：**
- subagent 的 context 必须包含该批次**所有源文-译文对的全文**，不能只给关键词或摘要——否则 subagent 无法判断语义质量
- 自动化扫描结果作为**补充**注入 subagent context，帮助 subagent 快速定位问题但不替代人工判断
- TB 状态为 New 的术语偏离 → soft flag；WorkingTB/Approved → 硬性报 Terminology Major
- subagent prompt 模板见 `references/lqe_subagent_prompt.md`
