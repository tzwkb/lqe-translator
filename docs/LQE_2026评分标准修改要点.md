# LQE 2026 评分标准修改要点

日期：2026-07-02

## 1. 背景

本文件基于 `LQE Report_2026模板（试行）.xlsx` 与当前 `lqe-translator` 评分实现对比整理，用于指导后续代码、报告模板和评估口径调整。

当前机制的核心文件：

- `/Users/spellbook/.codex/skills/lqe-translator/scripts/lqe_engine.py`
- `/Users/spellbook/.codex/skills/lqe-translator/scripts/lqe_calc.py`
- `/Users/spellbook/.codex/skills/lqe-translator/scripts/lqe_io.py`

新模板的核心 sheet：

- `LQE Error Severity`
- `LQE Error Definition`
- `Scorecard`

## 2. 总体结论

总分公式与当前实现基本一致，仍是：

```text
Score = MAX((1 - weighted_penalty / wordcount) * 100, 0)
```

阈值仍为 98，`Final score < Threshold` 为 FAIL，否则 PASS。

主要差异不在公式，而在三处：

1. 严重度分值改变：新模板 `Minor=2`，当前实现 `Minor=1`。
2. 类别权重大幅调整：新模板不再使用当前的 1.0 / 1.5 简化权重。
3. 分类体系不同：新模板仍使用 `Accuracy / Fluency / Verity / Design` 等旧父类；当前实现已对齐 MQM / ISO 5060 父维度，并新增了 `Audience appropriateness`。

## 3. 新模板评分参数

|---|---:|---:|
| Neutral | 0 | 0 |

| Major | 5 | 5 |
| Critical | 10 | 10 |

影响：轻微错误扣分翻倍。包含标点、拼写、轻微一致性、轻微自然度等问题的报告，按新模板会更容易掉分。

新模板更重视玩家/客户感知层面的自然度、文化适配、术语、项目风格和 locale；相对降低了语法、拼写、长度、Markup 以及普通准确性问题的权重。

## 4. 分类体系差异

### 4.1 父类命名差异

当前实现已将父类对齐到 MQM / ISO 5060：

- Terminology
- Accuracy
- Linguistic Conventions
- Style
- Locale Conventions
- Audience Appropriateness
- Design and Markup
- Other

新模板使用：

- Accuracy
- Fluency
- Terminology
- Style
- Locale convention
- Verity
- Design
- Other

建议：内部可以继续保留 MQM / ISO 5060 父维度，但导出 2026 模板时需要映射回模板父类。

### 4.2 `Audience appropriateness` 处理

当前实现有 `Audience appropriateness` 子类，新模板没有。

建议处理方式：

1. 内部保留 `Audience appropriateness`，便于评估时区分“准确但语域/受众不合适”的问题。
2. 2026 模板导出和计分时，将其折算到 `Unidiomatic` 或 `Company style`：
   - 若违反项目明文风格、角色语气或受众定位：映射到 `Company style`。
   - 若只是目标语表达自然度/语域不自然：映射到 `Unidiomatic`。
3. 如无法自动判断，默认映射到 `Unidiomatic`，因为新模板将其作为最高权重类别。

## 5. 必要代码修改点

### 5.1 新增评分 profile，避免破坏历史报告

不建议直接覆盖现有常量。建议新增评分档：

```text
legacy    当前机制
lqe2026   新模板机制
mqm       现有可选 MQM severity scale，可继续保留
```

涉及文件：

- `lqe_engine.py`
- `lqe_calc.py`
- `lqe_io.py`
- `finalize_job.sh`
- `aggregate_sheets.py`，如多 sheet 聚合报告也需要同口径

### 5.2 修改严重度分值

在 `lqe_engine.py` 中新增：

```python
SEVERITY_POINTS_LQE2026 = {
    "Neutral": 0,
    "Minor": 2,
    "Major": 5,
    "Critical": 10,
}
```

`lqe_calc.py` 增加参数：

```text
--scoring-profile legacy|lqe2026
```

当 `--scoring-profile lqe2026` 时，使用 `SEVERITY_POINTS_LQE2026`。

### 5.3 修改类别权重

在 `lqe_engine.py` 中新增：

```python
WEIGHTS_LQE2026 = {
    "Addition": 1.0,
    "Mistranslation": 1.0,
    "Omission": 1.0,
    "Untranslated": 1.0,
    "Inconsistency": 2.0,
    "Grammar": 0.5,
    "Punctuation": 0.5,
    "Spelling": 0.5,
    "Terminology": 2.0,
    "Unidiomatic": 3.5,
    "Company style": 2.0,
    "Locale convention": 2.0,
    "Culture specific reference": 2.5,
    "Length": 0.5,
    "Markup": 1.0,
    "Other": 0.5,
}
```

`Audience appropriateness` 不直接进入新模板权重表，应在计分前映射到 `Unidiomatic` 或 `Company style`。

### 5.4 更新父类映射

新增 2026 模板导出父类映射：

```python
CATEGORY_PARENT_LQE2026 = {
    "Addition": "Accuracy",

    "Locale convention": "Locale convention",
    "Culture specific reference": "Verity",
    "Length": "Design",
    "Markup": "Design",

```

### 5.5 保留强制 Major 规则

当前代码会强制：

- Terminology -> Major
- Untranslated -> Major
- Markup -> Major
- Length -> Major

新模板说明中同样写了这些类别 always major。建议保留当前强制逻辑。

注意：模板 Excel 本身不会自动强制严重度，依赖审校填写；我们的脚本强制更安全。

### 5.6 报告模板需要同步

当前 `lqe_io.py` 自建报告，而不是直接套用 `LQE Report_2026模板（试行）.xlsx`。如果要对齐客户模板，有两种方案：

方案 A：继续自建报告，但按新模板调整字段、权重、父类命名。

优点：改动较小，现有导出流程稳定。

缺点：视觉上不完全等同客户模板。

方案 B：读取客户 Excel 模板，填入 `Scorecard` 明细区，让 Excel 公式自动计算。

优点：客户看到的格式完全一致。

缺点：实现复杂，需要处理模板复制、公式保留、行数扩展、数据验证、样式和多 sheet 聚合。

建议先做方案 A，确认分数口径无误后再做方案 B。

## 6. 兼容性建议

### 6.1 默认值选择

若后续所有项目都必须采用 2026 标准：

```text
默认 --scoring-profile lqe2026
```

若历史项目仍需复现旧报告：

```text
默认 legacy，项目 profile.json 显式配置 scoring_profile: lqe2026
```

推荐第二种，比较稳。

### 6.2 profile.json 新增字段

建议项目档案支持：

```json
{
  "scoring_profile": "lqe2026",
  "threshold": 98
}
```

读取优先级：

```text
CLI 显式参数 > project profile > 默认值
```

### 6.3 输出报告标注

报告中必须明确显示：

- Scoring profile: LQE 2026
- Severity points: Neutral 0 / Minor 2 / Major 5 / Critical 10
- Threshold: 98
- Repeated errors excluded from weighted penalty

否则 PM 很难判断两个报告为什么分数不同。

## 7. 验证用例建议

### 7.1 单条错误扣分用例

准备固定 wordcount，例如 1000，逐类构造单条错误：

- 1 x Minor Punctuation：新模板扣 `0.5 * 2 = 1`，得分 99.90
- 1 x Major Mistranslation：新模板扣 `1 * 5 = 5`，得分 99.50
- 1 x Major Unidiomatic：新模板扣 `3.5 * 5 = 17.5`，得分 98.25
- 1 x Major Grammar：新模板扣 `0.5 * 5 = 2.5`，得分 99.75

### 7.2 Repeated 用例

同源同译、同类别同严重度重复两次：

- 第一条计分
- 第二条标 `repeated: true`
- 第二条不进入 weighted penalty

### 7.3 强制 Major 用例

输入 `Terminology/Untranslated/Markup/Length` 且 severity 填 `Minor`：

- 计算前应强制改为 `Major`
- 报告明细中也应显示 Major

### 7.4 Audience appropriateness 映射用例

构造一条 `Audience appropriateness`：

- 如违反项目 SG，映射到 `Company style`
- 如只是表达不自然，映射到 `Unidiomatic`
- 最终导出的 2026 报告不应出现模板没有的子类

## 8. 实施顺序建议

1. 在 `lqe_engine.py` 中新增 LQE 2026 severity points、weights、parent mapping。
2. 在 `lqe_calc.py` 增加 `--scoring-profile`，计算时按 profile 选择参数。
3. 在 `lqe_io.py` 报告生成中按 profile 输出权重表和父类名。
4. 在项目 `profile.json` 中支持 `scoring_profile`。
5. 更新 `finalize_job.sh` 和多 sheet 聚合脚本，保证 profile 参数贯穿全链路。
6. 添加单元测试覆盖 severity、weights、repeated、forced major、category mapping。
7. 用一个历史 job 双跑 `legacy` 与 `lqe2026`，人工确认差异符合预期。

## 9. 风险点

1. 新模板 `Unidiomatic=3.5` 权重极高，AI 审校如果过度报自然度问题，会显著压低分数。
2. `Grammar=0.5` 很低，旧机制下严重语法错误在新模板中扣分会变轻，需要评估时更谨慎判断是否应归到 `Unidiomatic` 或 `Mistranslation`。
3. `Audience appropriateness` 不在新模板中，直接导出会导致客户模板统计不到该类。
4. 如果只改 `lqe_calc.py` 不改 `lqe_io.py`，命令行分数和 Excel 报告里的权重展示会不一致。
5. 如果多 sheet 聚合未同步 profile，单 sheet 报告与总报告可能分数不一致。

## 10. 推荐决策

推荐采用：

```text
新增 scoring_profile=lqe2026，不删除 legacy。
项目 profile 控制默认评分口径。
内部保留 MQM/ISO 分类能力，导出 2026 报告时映射到客户模板分类。
```

这样既能接入新标准，也能回溯旧项目结果，风险最低。
