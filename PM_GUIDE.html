一、用途与准备

LQE Translator 是 Codex 中的译文质检 skill。它先运行机器预检，再由术语、准确性、语法和自然度检查模块报告问题；脚本校验局部修改、计算 LQE 分数并生成 Excel 交付文件。

必需Excel、CSV 或 TSV；明确原文列、译文列和项目语言轨，例如 nrc/zh-th
项目资料profile.json、confirmed_rules.md、风格指南、术语表和目标语言说明
运行环境macOS 或 Linux shell、Python 3；首次安装 openpyxl requests python-docx
运行选择默认首轮检查；只有明确要求时才启用自动迭代
两种保护：术语条目的 protected: true 表示不可修改；有明确 TM 精确匹配证据的段也可标记为受保护。受保护内容不修改、不计分。

二、手动启动与运行

在 skill 根目录运行。默认安装位置为 ~/.codex/skills/lqe-translator：

cd ~/.codex/skills/lqe-translator pip install openpyxl requests python-docx -q SCRIPTS="$PWD/scripts" JOB="$PWD/jobs/<任务名>"

用项目档案初始化，并把机器预检结果单独写入 errors_precheck.json：

python3 "$SCRIPTS/lqe_io.py" read --project "<game/source-target>" \ --input "<输入文件绝对路径>" --source-col "<原文列>" --target-col "<译文列>" \ --out "$JOB/state.json" python3 "$SCRIPTS/lqe_io.py" pre-check --state "$JOB/state.json" \ --out "$JOB/errors_precheck.json" python3 "$SCRIPTS/lqe_chunk.py" split --state "$JOB/state.json" \ --errors "$JOB/errors_precheck.json" --terms "$JOB/terms.json" \ --outdir "$JOB/chunks" --size 100

1初始化读取输入和项目档案，确认段数、固定词数及资料加载情况
▼
2确认受保护内容只根据客户资料、用户确认或明确的 TM 匹配证据设置
▼
3机器预检检查空译文、变量、标签、数字、长度、格式、术语和项目规则
▼
4专项检查子任务顶层只输出 {id, issues}；确认项写 needs_confirmation: true 和 edit: null；完整建议译文只由程序生成
▼
5结构校验、合并与评分四个必需模块必须覆盖全部段，脚本再生成建议译文
▼
6报告与导出生成标准 LQE 报告和建议稿；首轮结果不足时由 PM 决定是否继续

技术收尾命令固定为：

bash "$SCRIPTS/finalize_job.sh" "$JOB" <分块数> single

single 只生成首轮报告和建议稿；用户明确要求自动迭代时才改为 iterate。

三、自然语言 Agent 操作

复制下面的提示词，替换尖括号内容：

使用 lqe-translator 检查 <输入文件绝对路径>。 项目语言轨：<game/source-target>；原文列：<列名>；译文列：<列名>。 先读取项目背景、confirmed_rules.md、风格指南、术语表和目标语言说明。 只跑首轮。受保护内容必须保留，不修改用户原文件。 检查子任务顶层只提交 {id, issues}；无法安全局部修改的项设置 needs_confirmation: true、edit: null；完整建议译文由程序生成。 完成后给出段数、固定词数、分数、问题数、保护数，以及 *_lqe.xlsx 和 *_corrected.xlsx 的绝对路径；任何校验失败都要原样报告。

需要自动迭代时，单独增加：“本任务明确启用自动迭代，直到达到项目阈值或出现需要人工决定的问题。”

四、产出物

Excel 任务完成后，标准文件位于 jobs/<任务名>/：

文件用途
*_lqe.xlsx分数、逐段问题、建议译文、处理方式和历史记录，供 PM 与审校检查
*_corrected.xlsx保留原工作簿结构，只把通过校验的建议修改写入译文列

报告中的处理方式只使用“建议修改、需要人工确认、保持原译、已保护”。原输入文件不会被覆盖。

五、验收检查

输入与计数任务段数与有效输入一致；固定词数已记录，后续轮次不变化
模块覆盖四个必需模块覆盖全部分配 id；结构校验无缺失、重复或非法字段
保护规则受保护段的问题数和改动数均为 0，导出文本与原译一致
修改安全变量、标签、换行和受保护文本没有变化；需要人工确认的项没有自动写入
交付文件*_lqe.xlsx 与 *_corrected.xlsx 均存在且可打开；工作表、空行、列顺序和格式保持
结果说明首轮数据与重试数据分开报告；未达到阈值时不描述为通过

六、常见问题与恢复

初始化找不到项目？

确认语言轨写成 <game>/<source>-<target>，并检查该目录的 profile.json、必填语言字段和相对资源路径。

检查模块无法合并？

先运行 validate-checks --job "$JOB"。按报错补齐四个必需模块和全部 id，不要跳过失败继续导出。

某项不能安全修改？

保留原译，在报告中标记“需要人工确认”。不要把多个合理方案中的任一个直接写入建议稿。

运行中断？

保留 jobs/<任务名>/。分块超过 30 段时可从 *.ckpt.jsonl 断点继续，完成后再生成正式模块 JSON。

客户确认了新规则？

把明确结论写入当前语言轨的 confirmed_rules.md，下次检查前读取；不要把项目规则写进目标语言通用事实。
